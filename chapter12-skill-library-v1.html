Part 3 · 第 12 章 · 建立 AI 系统

建立你的 Skill 库
把经验沉淀成可复用资产——从反复提问到一次建库

每次遇到类似任务都重新写 Prompt，其实是在浪费时间。真正高效的人不是"更会提问"，而是把反复验证过的提问方式封装起来，变成随时可调用的模块。这一章讲的，就是如何把经验变成资产。

核心方法论：经验资产化 — Skill 是一个有标准输入、固定提示词逻辑、清晰输出格式的"AI操作模板"。建立 Skill 库不是收集 Prompt，而是把你在实际工作中反复验证过的好用做法，按场景整理成可调用的模块，让以后每次启动同类任务的时候不用重新思考，直接套用。
小节一

Prompt 和 Skill 有什么本质区别

认知基础 · 理解区别

用了一段时间 AI 之后，很多人会进入一种状态：手上攒了几十条"好用的 Prompt"，有些存在备忘录里，有些发在微信收藏，有些只记得"好像有一次我这样问效果特别好"——但下次遇到同类任务时，还是要重新思考、重新试错，找不到之前那个好用的版本。

这说明你积累的是 Prompt，而不是 Skill。两者的区别不是形式，而是结构和可复用程度。

Prompt 是临时提问，Skill 是可复用的结构化操作模块

Skill 的本质是一个有三个要素的模块：标准化的输入字段（你每次需要填写什么内容）、经过验证的提示词逻辑（告诉 AI 怎么处理、输出什么格式）、清晰的输出规格（结果是什么样的、有哪些部分）。有了这三个要素，它才能被复用——你只需要填写当次的输入，不需要每次都重新思考怎么问。

还有一个更直接的区分方式：如果你能把这套操作交给另一个同事，对方看完说明就能独立使用、得到差不多质量的结果——它才算 Skill。如果只有你自己知道怎么用，它还只是你脑子里的"感觉"。

小节二

什么值得变成 Skill——三个判断标准

筛选逻辑 · 什么值得沉淀

不是所有 Prompt 都值得封装成 Skill。封装是有成本的：你要整理结构、测试输入、规范输出格式。如果一个任务一年只做一次，花时间封装的收益远不如直接用。

判断一个 Prompt 是否值得升级为 Skill，有三个标准。满足其中两个就值得做：

①
高频重复
同类任务每月至少做2次以上，输入格式基本一致，不是一次性的特殊需求
②
已验证好用
这个 Prompt 在实际场景里用过、输出质量达到你的要求，不是第一次尝试
③
输入可标准化
任务所需的原材料是固定类型的（文字/表格/数据），不是每次都完全不同的内容
举例：哪些值得建成 Skill

✅ 值得：每周写工作汇报、每次收到竞品动态需要摘要整理、每次来新员工需要生成入职培训材料提纲、每月整理销售数据需要生成分析报告……这些任务结构固定、频率高、原材料格式稳定。

❌ 不值得：临时帮领导写一封特殊的感谢信、一次性的市场调研总结、处理特定客户投诉的定制化回复……这些任务每次情况差异太大，标准化之后反而会变得僵硬。

另一个实用的判断方式是看你有没有在对 AI 说过"上次那个很好用，但我忘了我是怎么问的了"。只要你出现过这种情况，那个"上次的好用做法"就是一个待沉淀的 Skill 候选。

💡 建议的筛选节奏

不要从零开始想"我应该建哪些 Skill"，而是先用两周，边用边记录。每次你觉得某个 AI 对话结果特别好、或者某次花了很长时间调试 Prompt 终于出了满意结果时，把它标记一下。两周后，从这些标记里挑频率最高的3-5个，先把它们变成 Skill。

小节三

一个 Skill 的完整结构长什么样

模板设计 · 标准格式

一个设计良好的 Skill，本质上是一张可填写的操作卡片。它需要让任何拿到它的人，都能在不和你沟通的情况下，独立完成同样质量的工作。这意味着它必须把所有"需要思考的部分"都事先设计好，使用者只需要填写当次的具体内容。

一个完整的 Skill 包含六个字段：

Skill 卡片模板 · 标准格式
Skill 名称
一句话说清楚这个 Skill 做什么，比如"周报生成器"或"竞品动态摘要提炼器"
适用场景
什么时候用它、用于什么工作，方便以后搜索和分类，如"每周五整理工作汇报时使用"
输入字段（每次使用前需要填写的内容）
【本周工作内容】：粘贴本周完成的事项（不需要格式化，原始记录即可）
【所属部门/岗位】：用于校准汇报口吻
【汇报对象】：直属上级 / 隔级汇报 / 跨部门
提示词主体（复制给 AI 的指令，含占位符）
你是一位职场写作助手。请根据以下信息，生成一份专业的工作周报。

【工作内容】：{本周工作内容}
【部门/岗位】：{所属部门/岗位}
【汇报对象】：{汇报对象}

请按以下结构输出：
1. 本周亮点（2-3点，每点1-2句话）
2. 主要进展（分项列举，附完成度）
3. 遇到的问题/风险（如有）
4. 下周计划（3-5项）

风格要求：简洁专业，避免空话，数字优先。
预期输出格式
结构化文字，4个部分，总字数300-500字，无需 Markdown 标记，可直接粘贴使用
注意事项 / 已知局限
如本周工作内容较少（不足3项），AI 可能会"水"出不实际的进展描述，需要手动删减；不适用于高度保密的项目汇报场景

上面这个结构看起来有点繁琐，但只需要第一次建的时候认真填，之后每次用只需要填"输入字段"那一栏。关键是提示词主体里的占位符——那些用 {} 括起来的内容，就是每次需要替换的变量，其余部分完全不需要改动。

⚠️ 常见错误：Skill 写得太死

很多人建 Skill 时会把所有细节都写进提示词，比如"本周是年度绩效季，请特别强调……"——这就把一个通用 Skill 变成了一个一次性用品。好的 Skill 设计原则是：固定的部分（逻辑、结构、风格）放进提示词主体，变化的部分（具体内容、当次背景）留作输入字段。

小节五

Skill 库怎么组织——三层结构，不要只用一个文档

组织方式 · 库的结构

很多人把 Skill 集中存在一个超长的备忘录文档里。这在前5个 Skill 的时候没问题，但当你积累到20个以上时，找起来会越来越慢，而且没有结构的文档让你很难知道"自己究竟有多少 Skill、覆盖了哪些场景"。

推荐用三层结构来组织你的 Skill 库：

1
第一层：Skill 索引表（一张表格）
所有 Skill 的总目录，每行一个 Skill，包含：名称、适用场景标签、使用频率、链接/位置。这层的作用是让你在5秒内找到想用的 Skill，不需要翻内容细节。可以放在飞书表格、Notion、甚至 Excel 里。
2
第二层：Skill 卡片（每个 Skill 一张卡）
按上一小节的六字段结构填写的完整 Skill 文档。可以是飞书文档的子页面，也可以是 Notion 的 database record，甚至是一个文件夹里的 .txt 文件——格式不重要，重要的是结构完整、信息齐全、不需要靠记忆补全。
3
第三层：快捷启动（你的常用工具里）
对于每周使用超过3次的高频 Skill，把它的提示词主体直接存在你最常用的输入位置——比如 Claude 的项目 Instructions、ChatGPT 的 Custom Instructions、或者一个桌面快捷文件。这层的目标是"开始任务后立即可用"，不需要打开任何额外文档。

小节六

从一个好用的 Prompt 到一个 Skill——实操四步

实操流程 · 四步升级

假设你上周用 AI 做了一次竞品分析，结果非常好，现在想把这次的做法固化成一个可复用的 Skill。走一遍升级流程：

1
回溯这次好结果的关键原因
找到你上次的对话记录，看一看：是提示词里哪些部分起了关键作用？是"告诉 AI 你想对比哪几个维度"，还是"要求 AI 用表格输出"，还是"先给了一个案例告诉 AI 你想要什么风格"？把这些关键要素标记出来，这是提炼提示词主体的原材料。
2
识别变量部分，提炼为输入字段
把原来提示词里那些每次都会变化的内容（竞品名称、分析维度、目标读者）替换成占位符 {竞品名称}、{分析维度}。剩下固定不变的部分（AI的角色、输出格式要求、思考框架）就是提示词主体。这一步通常很机械，但是把"一次性 Prompt"和"可复用 Skill"拆开的关键动作。
3
用不同的输入测试两到三次
用这个 Skill 框架做两到三次不同的竞品分析（换不同的产品、换不同的分析维度），检查输出质量是否稳定。如果某一次效果明显变差，找到原因：是输入字段不够清晰、还是提示词主体有某个约束条件太宽松？根据测试结果调整。
4
填写完整的 Skill 卡片，存入库
按照上一小节的六字段格式，填写完整的 Skill 卡片，包括"注意事项/已知局限"（这个字段很重要，记录你在测试中发现的边界情况）。存入 Skill 库，更新索引表。整个过程通常只需要15-20分钟。

我把这些整理成 Skill 卡片：

上面这个对话说明了一件事：你不需要自己从零开始整理 Skill，可以直接让 AI 帮你从已有的好用对话中提炼结构。这是一个被很多人忽视的技巧——AI 不只是执行工具，它也可以帮你整理和沉淀你用 AI 的经验本身。

小节七

Skill 库的维护——不要建完就放着不管

维护机制 · 持续迭代

Skill 库最常见的死法，不是"没有建起来"，而是"建起来之后慢慢腐烂"——里面的 Skill 越来越老、越来越不符合实际工作需求，但没有人更新，最后变成了一个没人看的文档死角。

机制一：使用时随手标记。每次用完一个 Skill，花30秒做一个标记：这次效果好不好？有没有遇到新的边界情况？是否需要调整某个输入字段？不需要立刻改，只需要记下来。最简单的方式是在 Skill 卡片文档末尾加一行"[2026-03-28 反馈：竞品数量为4个时表格显示有问题，考虑加一条判断提示]"。

机制二：每月固定复审。每个月花30分钟过一遍 Skill 索引，做三件事：把上个月收集到的反馈逐条处理；把三个月没有用过的 Skill 标记为"待确认"（也许那个工作场景变了）；把最近发现的新好用 Prompt 整理进库。30分钟不多，但这是 Skill 库保持活力的核心动作。

动作 频率 时间成本 作用
使用后标记反馈 每次使用时 30秒 积累改进原材料，不让问题沉没
处理反馈、更新 Skill 每月一次 20分钟 让 Skill 保持与实际工作同步
扫描沉睡 Skill 每月一次 5分钟 识别不再适用的 Skill，避免库腐烂
新增 Skill 入库 按需（通常每月1-3个） 15-20分钟/个 库的持续增长，覆盖更多场景
💡 一个有用的习惯：Skill 版本号

当你对一个 Skill 做了较大改动时，给它加一个版本标记，比如"v1.2（2026-03更新：加入对报告读者的风格适配逻辑）"。这有两个好处：一是让你随时知道这个 Skill 上次是什么时候改的，二是万一新版本效果反而不如旧版，你能回溯。不需要保存完整历史版本，只需要在卡片里加一行版本说明即可。

小节八

Skill 的三种使用模式——从手动调用到半自动触发

使用模式 · 三种方式

Skill 建好之后，有三种使用方式，复杂度和效率递增。你不需要一开始就追求最高级的方式，按实际需求选择就好。

三种 Skill 使用模式——按使用频率和自动化需求选择

大多数人从模式一开始就足够了。当你发现某个 Skill 每天都要用，打开文档复制粘贴开始觉得麻烦的时候，升级到模式二——在 Claude 或 ChatGPT 里建一个专项 Project，把这个 Skill 的提示词设置为 Project Instructions，以后只要打开这个 Project 开始对话，Skill 就自动激活。

模式三是更进阶的用法，当一个 Skill 不再需要你手动启动、而是由某个事件自动触发时（比如每次收到新简历邮件就自动运行简历解析 Skill），它就从"手动工具"变成了"工作流节点"。这是第13章的主题——把多个 Skill 串起来，变成一条自动化工作流。

🔑 核心洞察：Skill 是 Workflow 的积木，也是可移植的资产

你现在建的每一个 Skill，本质上都是未来工作流里的一块积木。当你有了足够数量的、质量稳定的 Skill，把它们按顺序串起来就变成了工作流。这就是为什么 Skill 库是 Part 3 的起点——没有好的 Skill，工作流只是空架子；有了稳定的 Skill，工作流就有了可靠的执行单元。

还有一件事值得提前知道：2025年12月，Agent Skills 格式成为跨平台开放标准，Claude Code、GitHub Copilot、VS Code、Cursor 等主流工具都已原生支持。这意味着你现在积累的 Skill 文档，不绑定任何单一工具。你建的是判断标准本身，而不是某个平台的配置文件。

小节九

你的第一个 Skill 库——三周落地路线

行动计划 · 从零开始

理解了方法论之后，很多人还是会陷入"不知道从哪里开始"的状态。下面是一条三周建立 Skill 库的最简化路线，每一步都有具体的交付物。

第一周
积累观察，不要急着建
这一周正常使用 AI，但每次用完做一件事：在备忘录里记一行"[日期] 做了什么 + 效果好不好 + 关键原因是什么"。不需要整理格式，流水账记法就行。到周末，你大概会有5-10条记录。
第二周
从记录里提炼第一批 Skill
从上周的记录里挑 3 个"效果好、还会再用"的案例，按六字段格式各整理一张 Skill 卡片。同时建好索引表（一张简单的飞书/Excel 表格），把这3个 Skill 录进去。本周末，你有了 3 张 Skill 卡片 + 1 张索引表——这是你的 Skill 库的最小可行版本。
第三周
测试、迭代，培养使用习惯
这一周正式启用你的 Skill 库。每次遇到之前建了 Skill 的场景，优先从库里调用而不是重新想。注意：第一次用 Skill 时可能会觉得"填输入字段比直接说更麻烦"——这是正常的，第三次之后就会感受到差异。同时继续积累，把新发现的好 Prompt 加入候选队列。

三周之后，你不需要有完美的 Skill 库，需要的只是两样东西：一个建立了基础结构的库（哪怕只有5-10个 Skill），以及一个使用的习惯（遇到老任务先看库，不是先想提示词）。从这里开始，Skill 库会自然随着你的工作积累而增长，不需要专门"抽时间来建"。

✅ 本章交付物检查

完成这一章之后，你应该有：① 一份包含3-5个 Skill 的索引表（有名称、场景标签、使用频率）② 对应的 Skill 卡片（六字段填写完整）③ 至少一个 Skill 被配置为 AI 项目的预置提示词（模式二）④ 一个简单的每月复审提醒（日历备注即可）。以上四项是本章的最小交付物，也是进入第13章（工作流）的前置条件。

小节十

为什么你的同事还在"收藏 Prompt"而你在"运营资产"

视角对比 · 长期竞争力

大多数人对 AI 能力积累的理解，停留在"攒 Prompt"的阶段——在小红书收藏了200个 Prompt、在微信收藏里有一个"AI神器合集"文档，但这些内容90%从来不会再看第二遍，更不会被系统性地用进工作。

Skill 库和 Prompt 收藏的根本差异，不是整理得好不好，而是对"AI能力"的理解深度不同。看一下这张对比表：

维度 Prompt 收藏 Skill 库
来源 别人分享的 + 偶尔自己想的，主要是输入型积累 自己实际使用并验证过的，完全是输出型积累
结构 无固定格式，靠关键词和记忆定位 六字段标准结构，有索引，可被他人理解
可复用程度 低，每次用还需要思考怎么适配当下情境 高，填写输入字段即可使用，不需要重新思考
增长方式 随机积累，越积越多越难用 主动沉淀，越用越精简越好用
可分享性 分享出去对方不一定懂怎么用 有说明文档，可以直接移交给同事使用
核心价值 降低"想提示词"的门槛 把你的工作经验和判断力固化成可调用的系统

最后一行是关键。Prompt 解决的是"怎么更好地问 AI"，而 Skill 库解决的是"怎么把你在工作中积累的业务理解转化为可持续调用的资产"。前者让你用得更流畅，后者让你建立起真正意义上的个人 AI 竞争力。

三年后，可能每个人都会用 AI，会写 Prompt 不再是优势。但拥有一个根植于自己业务场景、经过反复验证、持续维护的 Skill 库——这是你自己的经验在 AI 系统里的沉淀，它无法被简单地复制或替代。

✅ 本章核心结论

Skill 是结构化的可复用操作模块，不是随手记下来的提示词。建立 Skill 库的本质是经验资产化——把你在实际工作中反复验证过的好做法，按标准格式整理成随时可调用的模块。从第一周记录观察、第二周提炼前三个 Skill、第三周开始实际使用，三周之后你就拥有了一个有活力的个人 AI 资产库。这是接下来搭工作流的基础。

📚 本章小结

Prompt 解决一次，Skill 解决一类。把经验沉淀成 Skill 的核心动作是：识别高频可复用场景 → 提炼标准输入字段和提示词主体 → 填写六字段卡片 → 建立三层库结构 → 用使用时标记 + 每月复审的机制让库保持活力。Skill 库是你在 AI 时代最值得建立的个人资产之一，因为它不只是一个工具列表，而是你的业务判断和工作经验在系统里的固化形式。
